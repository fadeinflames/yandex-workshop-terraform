Зачем Infrastructure as Code?

Без IaC

  • Кликаем в консоли
  • "А кто это создал?"
  • "Какие настройки у прода?"
  • Документация устарела на следующий день

С IaC

  • Код в git = документация
  • Code review для инфраструктуры
  • Воспроизводимость: terraform apply
  • История всех изменений

Что такое Terraform?

  • Декларативный — описываем ЧТО хотим, не КАК
  • Мульти-облачный — AWS, GCP, Yandex Cloud
  • State — помнит что создал
  • Plan — показывает что изменится ДО применения

Облакоресурсы

Основные концепции

Provider

Подключение к облаку. Плагин, который умеет общаться с API.

Resource

Ресурс, который создаем: VM, сеть, диск.

Variable

Входной параметр. Как .env файл.

Output

Выходное значение. Как return в функции.

Data Source

Чтение того, что уже существует в облаке.

State

"Память" Terraform. Помнит что создал.

Provider — подключение к Yandex Cloud

terraform {
  required_providers {
    yandex = {
      source = "yandex-cloud/yandex"
    }
  }
}

provider "yandex" {
  token     = var.yc_token
  cloud_id  = var.yc_cloud_id
  folder_id = var.yc_folder_id
  zone      = "ru-central1-a"
}

Resource — создаем ресурсы

resource "yandex_vpc_network" "workshop" {
  name = "workshop-network"
}

resource "yandex_vpc_subnet" "workshop" {
  name           = "workshop-subnet"
  network_id     = yandex_vpc_network.workshop.id  # ссылка!
  v4_cidr_blocks = ["10.0.1.0/24"]
}

Terraform сам определяет порядок создания по ссылкам между ресурсами.

Виртуальная машина

resource "yandex_compute_instance" "demo" {
  name        = "demo-vm"
  platform_id = "standard-v3"

  resources {
    cores  = 2
    memory = 2
  }

  boot_disk {
    initialize_params {
      image_id = "fd8jr9omc57n4c6hev20"
    }
  }

  network_interface {
    subnet_id = yandex_vpc_subnet.workshop.id
    nat       = true   # внешний IP
  }
}

Рабочий цикл

terraform init      # 1. Скачать провайдер
terraform plan      # 2. Посмотреть что будет
terraform apply     # 3. Применить (спросит yes)

Символы в plan

СимволЗначение
+Ресурс будет создан
-Ресурс будет удален
~Изменение на месте (update in-place)
-/+Пересоздание (destroy + create)
<=Чтение (data source — вычитывает данные из облака)

Задание 1: Веб-сервер с nginx 15 мин

Возьмите готовый файл из students/block-1/.
Добавьте cloud-init чтобы nginx ставился автоматически.
По внешнему IP должна открываться страница.

user-data в metadata — скрипт, который выполняется при первом запуске VM. Формат: YAML с заголовком #cloud-config.

В network_interface нужно nat = true — это включает NAT и выдает внешний IP.

metadata = {
  ssh-keys  = "ubuntu:${file("~/.ssh/id_rsa.pub")}"
  user-data = <<-EOF
    #cloud-config
    package_update: true
    packages:
      - nginx
    runcmd:
      - systemctl enable nginx
      - systemctl start nginx
  EOF
}

Изменение инфраструктуры

Terraform не пересоздает все с нуля!

resource "yandex_compute_instance" "demo" {
  allow_stopping_for_update = true  # разрешаем stop/start

  resources {
    cores  = 2
    memory = 4   # было 2, стало 4
  }
}
terraform plan
# ~ yandex_compute_instance.demo
#     memory: 2 -> 4

Update in-place — меняет параметр без пересоздания VM.

Смена CPU/RAM требует стопа VM — нужен allow_stopping_for_update = true.

Но если сменить image_id — будет -/+ пересоздание!

Outputs

output "vm_external_ip" {
  value = yandex_compute_instance.demo
          .network_interface.0.nat_ip_address
}

output "ssh_command" {
  value = "ssh ubuntu@${...nat_ip_address}"
}
terraform output              # все outputs
terraform output -raw ssh_cmd # одно значение (для скриптов)

Outputs — как return в функции. Можно передавать между модулями и в CI/CD.

Lifecycle

lifecycle {
  # Сначала создать новый, потом удалить старый
  create_before_destroy = true

  # Запретить удаление (защита от terraform destroy)
  prevent_destroy = true

  # Игнорировать внешние изменения
  ignore_changes = [labels, metadata]
}
